# RFC/RFP: MVP обработки входящих звонков на Laravel + Filament

## Вводные

Этот MVP строится на Laravel 13 и Filament 5. Входящий звонок создает запись в `calls`, после чего в Redis-очередь отправляется `ProcessIncomingCallJob`. Система работает под нагрузкой с несколькими параллельными воркерами, из-за чего возникают риски гонок при назначении операторов и повторной отправки событий во внешнюю телефонию.

Предлагается MVP-архитектура на Laravel Queue с транзакционным назначением оператора, паттерном Outbox для внешнего HTTP и базовой операционной панелью на Filament для ручного контроля проблемных кейсов.

## Назначение и цели

### Почему это нужно сейчас

- Нельзя допускать двойного назначения одного звонка разным операторам.
- Нельзя полагаться на exactly-once от внешней телефонии.
- Нужен предсказуемый механизм ретраев и расследования ошибок в production.

### Цели RFC

- Обеспечить конкурентно-безопасное назначение звонка оператору при параллельной обработке.
- Обеспечить идемпотентную отправку события `call.assigned` во внешнюю телефонию.
- Обеспечить управляемое восстановление после ошибок (retry/backoff/failed).
- Добавить операционный интерфейс в Filament для мониторинга и ручных действий.

### Non-goals (вне MVP)

- Сложная маршрутизация операторов по навыкам/приоритетам/очередям.
- Переход на Kafka/Event Bus.
- Глубокая BI-аналитика, кроме базовых метрик состояния.

## Заинтересованные стороны

- Backend-разработчики (Laravel): реализация бизнес-логики, очередей, транзакций.
- DevOps/SRE: эксплуатация воркеров, мониторинг, алерты.
- Support/Operations: работа с инцидентами через Filament.
- Product/Team Lead: приемка бизнес-логики назначения и SLA обработки.

## Составляющие и особенности

### 1. Данные и состояния

#### Таблица calls

- Базовые поля: `id`, `phone`, `status`, `client_id`, `operator_id`, `assigned_at`, `dispatched_at`, `attempts_assign`, `last_error`, `created_at`, `updated_at`.
- Статусы MVP: `new`, `assigned`, `dispatched`, `failed`.
- Индексы:
	- `index(status, created_at)`
	- `index(phone)`

#### Таблица operators

- Поля: `id`, `available`, `last_call_at`, `active` (опционально), timestamps.
- Индекс: `index(available, last_call_at)`.

#### Таблица outbox_events

- Поля: `id`, `type`, `aggregate_type`, `aggregate_id`, `payload_json`, `status`, `attempts`, `next_retry_at`, `last_error`, timestamps.
- Уникальный индекс: `(type, aggregate_type, aggregate_id)` для защиты от дублей события.

### 2. Очереди и jobs

#### Очередь calls_assign

- Назначение оператора и фиксация состояния звонка.
- Job: `ProcessIncomingCallJob`.

#### Очередь telephony_dispatch

- Доставка outbox-события во внешнюю телефонию.
- Job: `DispatchOutboxEventJob`.

### 3. Конкурентная безопасность

- Назначение выполняется только в DB-транзакции.
- Звонок читается с блокировкой `FOR UPDATE`.
- Оператор выбирается из `available=true` с блокировкой строки и при наличии поддержки СУБД через `SKIP LOCKED`.
- Смена состояния звонка происходит атомарно: `new -> assigned`.

### 4. Идемпотентность

- В `ProcessIncomingCallJob` повторная обработка звонка со статусом отличным от `new` завершает job без побочных эффектов.
- Outbox-событие `call.assigned` уникализируется индексом по агрегату.
- Во внешнюю телефонию передается `Idempotency-Key` в формате `call:{call_id}:assigned` (если поддерживается API).

### 5. Ретраи и обработка ошибок

- Для отсутствия доступного оператора: контролируемый retry с backoff и jitter.
- Для временных ошибок телефонии (timeout/5xx): retry через `next_retry_at` с экспоненциальной паузой.
- Для невосстанавливаемых ошибок (часть 4xx): перевод события в `failed`, фиксация `last_error`.

### 6. Логирование и наблюдаемость

- Структурные логи с ключами: `call_id`, `operator_id`, `outbox_event_id`, `job`, `attempt`.
- Метрики MVP:
	- latency `new -> assigned`
	- latency `assigned -> dispatched`
	- количество `failed` событий outbox
	- backlog по очередям

### 7. Filament админ-контур

#### CallsResource

- Просмотр звонков: статус, оператор, клиент, timestamps, ошибки.
- Фильтры: статус, период, оператор.
- Действия: retry dispatch, mark failed, reset to new (с аудит-логом).

#### OperatorsResource

- Просмотр доступности операторов.
- Действия: force release / toggle available (с указанием причины в audit trail).

#### OutboxEventsResource

- Просмотр событий: статус, попытки, `next_retry_at`, ошибка.
- Действия: retry now, mark failed, view payload.

## Функциональные требования

- Каждый звонок может быть назначен не более чем одному оператору одновременно.
- При повторном запуске job не должно происходить повторного назначения.
- Событие назначения должно быть отправлено во внешнюю систему хотя бы один раз, с контролем дублей со стороны нашей системы.
- Операторы должны иметь возможность вручную переотправить/исправить проблемный кейс из Filament.

## Нефункциональные требования

- Корректность при параллелизме воркеров.
- Предсказуемое поведение при временной недоступности телефонии.
- Трассируемость всех критичных переходов состояния.
- Поддерживаемость и простота эксплуатации.

## Технические и архитектурные детали

- Laravel Queue (Redis) для асинхронной обработки.
- Транзакционная граница: только DB-операции; внешние HTTP вне транзакции через Outbox.
- Рекомендуемая БД: MySQL 8+ или PostgreSQL (для полноценных конкурентных примитивов).
- Деплой и масштабирование воркеров без изменения бизнес-логики за счет row-level locking.

## Критерии приемки (checklist)

- [ ] При конкурентном запуске 2+ воркеров один и тот же оператор не назначается двум звонкам одновременно.
- [ ] Повторный запуск `ProcessIncomingCallJob` для уже `assigned`/`dispatched` звонка не меняет данные.
- [ ] Для одного `call_id` создается не более одного outbox-события `call.assigned`.
- [ ] При 5xx/timeout телефонии событие переходит в retry-модель с backoff, без потери контекста.
- [ ] При невосстанавливаемой ошибке событие переходит в `failed`, ошибка доступна в Filament.
- [ ] В Filament доступны действия `retry`, `force release`, `mark failed` и они корректно меняют состояние.
- [ ] Добавлены интеграционные тесты на конкурентность и идемпотентность.

## Дорожная карта

1. Подготовить миграции и индексы (`calls`, `operators`, `outbox_events`).
2. Внедрить транзакционный `ProcessIncomingCallJob` с блокировками строк.
3. Внедрить `DispatchOutboxEventJob` и retry-политику доставки.
4. Настроить структурные логи и базовые метрики/алерты.
5. Реализовать Filament Resources и безопасные admin actions.
6. Добавить интеграционные тесты на гонки, ретраи и идемпотентность.
7. Провести нагрузочную проверку и зафиксировать пороги SLA.

## Риски и ограничения

- Некорректная работа при отсутствии поддержки нужных SQL-блокировок в конкретной версии СУБД.
- Непредсказуемое поведение внешнего API телефонии при отсутствии идемпотентных ключей.
- Ручные действия в админке могут нарушить инварианты без audit trail и валидаций.

## Альтернативные варианты

### Вариант A: прямой HTTP в `ProcessIncomingCallJob`

Плюсы:

- Проще в реализации на старте.

Минусы:

- Сложнее обеспечить идемпотентность и управляемые ретраи.
- Выше риск частично-зафиксированных состояний.

Компромиссы:

- Быстрый запуск, но высокий операционный риск.

### Вариант B: транзакция + Outbox (рекомендуемый)

Плюсы:

- Консистентность внутреннего состояния.
- Контролируемая доставка во внешнюю систему.
- Прозрачная эксплуатация через отдельные состояния outbox.

Минусы:

- Больше сущностей и кода.

Компромиссы:

- Умеренная архитектурная сложность ради production-надежности.

### Рекомендация по выбору

Выбрать Вариант B как базовый для MVP в production.

## Открытые вопросы

- Нужно ли освобождать оператора только по webhook завершения звонка или нужен fallback-процесс по таймауту?
- Какие коды 4xx телефонии считать невосстанавливаемыми на уровне бизнес-логики?
- Какой SLA по времени назначения (`new -> assigned`) и доставки (`assigned -> dispatched`) фиксируем для алертов?

## Итого и рекомендации

Реализация MVP через транзакционное назначение + Outbox + Filament дает практичный баланс между скоростью внедрения и надежностью под параллельной нагрузкой. Такой подход закрывает ключевые риски текущего процесса (гонки, дубли, неуправляемые ошибки) и формирует основу для дальнейшего масштабирования без архитектурного переписывания.
